# 基于 `06-stock-bi-agent` 代码的问题解答

## 1. 什么是前后端分离？

前后端分离是一种软件架构设计模式，将应用程序的**前端**（用户界面与交互逻辑）与**后端**（业务逻辑、数据处理、API 服务）拆分为两个独立的子系统，并通过标准接口（如 RESTful API、WebSocket）进行通信。

在 `06-stock-bi-agent` 项目中，通常：

- **前端**：使用 Vue/React 等框架构建的聊天界面，负责展示对话、接收用户输入、调用后端接口。
- **后端**：基于 Python（如 FastAPI/Flask）或 Node.js 提供 `/chat`、`/history` 等 HTTP 端点，负责处理股票数据查询、调用大模型、管理对话历史。

**核心优势**：
- **独立开发与部署**：前端和后端可并行开发、分别部署（如前端部署在 CDN，后端部署在云服务器）。
- **技术栈解耦**：前端专注 UI/UX，后端专注业务逻辑与模型调用。
- **扩展性好**：前端可对接多个后端服务（如额外接入金融数据 API），后端可同时服务 Web、App、小程序等客户端。

## 2. 历史对话如何存储，以及如何作为大模型的下一次输入？

### 2.1 历史对话的存储方式（含短期与长期存储）

在 `06-stock-bi-agent` 中，历史对话通常采用**分层存储**策略：

- **短期存储（会话级）**  
  使用内存缓存（如 Redis）或后端服务的内存字典，以 `session_id` 为键，存储最近 N 条对话（例如最近 20 条消息）。示例结构：
  ```json
  {
    "session_id": "abc123",
    "messages": [
      {"role": "user", "content": "贵州茅台今日股价是多少？", "timestamp": "2026-04-24T10:00:00Z"},
      {"role": "assistant", "content": "贵州茅台今日收盘价 1680 元...", "timestamp": "2026-04-24T10:00:05Z"}
    ]
  }

- **长期存储（持久化）**
  使用关系型数据库（SQLite、PostgreSQL）或 NoSQL（MongoDB），将每条消息落表，字段包括：id、session_id、role、content、create_time、metadata（如股票代码、tool_calls 信息）。这样支持跨会话的历史回溯与数据分析。

### 2.1 如何将历史作为大模型的下一次输入（含长期存储的使用）
大模型本身是无状态的，需要在每次请求中显式传入对话历史。长期存储在这里的作用是：当用户的会话跨天或短期缓存失效时，系统可以从数据库加载完整的过往消息。

- **读取历史** 
  根据 session_id 先从 Redis 短期缓存中获取最近的对话；如果缓存未命中或需要更早的记录，则回退到长期存储（数据库） 查询。通常取最近 K 条消息（K 受限于模型的上下文窗口，例如 4096 tokens）。
