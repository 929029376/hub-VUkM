### 1. 什么是前后端分离？

前后端分离是一种软件架构模式，将用户界面（前端）与业务逻辑、数据存储（后端）解耦，两者通过定义好的接口（通常是 HTTP RESTful API）进行通信。

在本项目中，这一模式体现得非常清晰：

- 前端（Streamlit 侧） ：代码主要集中在 demo/chat/chat.py 等文件中。它只负责：
  - 构建用户界面（展示聊天框、股票图表等）。
  - 维护用户在浏览器的会话状态（如 st.session_state 中的聊天记录）。
  - 通过 requests 库向后端发送 HTTP 请求。例如代码中的 requests.post("http://127.0.0.1:8000/v1/chat/") 。
- 后端（FastAPI 侧） ：代码集中在 routers/chat.py 和 services/chat.py 。它不关心页面长什么样，只负责：
  - 接收前端的请求数据。
  - 处理核心业务逻辑（例如调用 OpenAI 接口、操作数据库、与 MCP 服务器交互）。
  - 将结果以 JSON 格式或数据流（SSE）的形式返回给前端。
    优点 ：前端和后端可以独立开发和部署；如果未来想把前端从 Streamlit 换成 Vue/React，后端的 FastAPI 代码完全不需要修改。

### 2. 历史对话如何存储，以及如何将历史对话作为大模型的下一次输入？

###### 在本项目中，历史对话的存储和输入是一个双重维护机制： 前端状态缓存 + 后端数据库持久化 。

###### 第一步：历史对话的存储机制 历史记录被同时存储在两端：

前端临时存储 (Session State) ：
在 demo/chat/chat.py 中，使用 Streamlit 的 st.session_state.messages 列表来临时存储当前页面的聊天记录。当用户刷新页面或进入已有的对话时，前端会调用后端的 /v1/chat/get 接口拉取历史记录，并填充到这个列表中以供页面渲染。

后端双数据库持久化存储 ：
在 services/chat.py 的 chat 方法中，对话被存入了两个地方：

业务关系型数据库 ( SessionLocal , ORM映射表 ChatSessionTable & ChatMessageTable )：用于在前端展示“历史对话列表”，以及将用户问题( user_question )作为标题保存。

大模型专属状态数据库 ( AdvancedSQLiteSession )：存放在 ./assert/conversations.db 中。这是 openai-agent 框架专门用来维护大模型上下文状态的数据库。 

###### 第二步：如何将历史对话作为大模型的下一次输入 这部分主要依赖于后端的 AdvancedSQLiteSession 和 OpenAI Agent 框架。

关联 Session ID ：
每次对话都有一个唯一的 session_id 。当用户在前端发送新消息时，会将这个 session_id 连同 content 一起发送给后端 API ( demo/chat/chat.py:L104 )。

加载历史上下文 ：
在后端的 services/chat.py 中，代码实例化了 AdvancedSQLiteSession ：
```
session = AdvancedSQLiteSession(
    session_id=session_id, # 与 系统中的对话id 关联
    db_path="./assert/conversations.db",
    create_tables=True
)
``` 这个 session 对象会被传递给大模型代理（Agent）。

注入给大模型 ：
底层的大模型代理（Agent 框架）在收到新的用户输入时，会自动通过这个 AdvancedSQLiteSession 对象从 conversations.db 数据库中读取该 session_id 对应的所有历史消息（包含 System Prompt、之前的 User 提问和 Assistant 回答）。
随后，框架会将这些历史消息与最新的一条问题组装成一个完整的 Message 列表（即类似 [{"role": "system", "content": "..."}, {"role": "user", "content": "历史问题"}, {"role": "assistant", "content": "历史回答"}, {"role": "user", "content": "新问题"}] 的格式），一并发送给 OpenAI / Qwen 接口，从而让大模型“记住”之前的聊天内容并给出连贯的回复。
