# 前后端分离与历史对话管理详解（基于项目代码）

## 1. 什么是前后端分离？

前后端分离是一种软件架构模式，将**前端（用户界面）**与**后端（业务逻辑、数据处理）**独立开发、部署，通过标准化的接口（如 HTTP API、WebSocket）进行通信。

### 在项目中的体现

- **前端**：`Streamlit` 应用（`./demo/` 目录）负责界面展示和用户交互。它不直接操作数据库或调用大模型，而是通过 HTTP 请求调用后端 API。
- **后端**：`FastAPI` 服务（`main_server.py` 及 `./routers/`、`./services/`）提供 RESTful API，处理聊天、股票、用户等业务逻辑，并负责数据存储与外部模型调用。

# 历史对话的存储与作为大模型输入的方法详解

## 1. 历史对话如何存储？

### 存储位置
历史对话存储在后端数据库中，使用了 SQLite 数据库（如 `conversations.db` 或 `sever.db`）。

### 存储流程

1. **会话创建**  
   前端调用 `/v1/chat/init` 接口，后端生成唯一的 `session_id`，并在数据库中创建一条会话记录。

2. **消息保存**  
   每次用户发送消息，后端收到请求后：
   - 将用户的消息内容、角色（`user`）、时间戳等存入消息表，关联当前的 `session_id`。
   - 调用大模型获得回复后，再将助手的消息（角色 `assistant`）同样存入数据库。

3. **历史加载**  
   当用户再次进入同一个会话（例如刷新页面或从历史列表点击），前端调用 `/v1/chat/get?session_id=xxx`，后端根据 `session_id` 查询该会话下的所有消息（按时间升序），返回给前端渲染。

# 后端初始化会话接口（示例）
def request_session_id() -> str:
    response = requests.post("http://127.0.0.1:8000/v1/chat/init")
    return response.json()["data"]["session_id"]

# 前端加载历史消息
if "session_id" in st.session_state.keys() and st.session_state.session_id:
    data = requests.post("http://127.0.0.1:8000/v1/chat/get?session_id=" + st.session_state['session_id']).json()
    for message in data["data"]:
        st.session_state.messages.append({"role": message["role"], "content": message["content"]})
